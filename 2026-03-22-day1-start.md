完成了3DGS渲染器的第一步：加载PLY文件并在Unity中显示点云。过程中涉及几个核心概念，需要深入理解。

## 为什么需要4×4矩阵而不是3×3

3×3矩阵只能表示旋转、缩放、剪切这些线性变换，但无法表示平移。因为线性变换必须保持原点不变，而平移会移动原点。

数学上，旋转矩阵乘以原点(0,0,0)还是(0,0,0)，无法实现"移动到(5,0,0)"这样的操作。

4×4齐次坐标的解决方案：把3D点(x,y,z)扩展为(x,y,z,w)，其中w=1。这样平移可以写成矩阵乘法：

```

| 1 0 0 tx |   | x |   | x + tx |
| 0 1 0 ty | × | y | = | y + ty |
| 0 0 1 tz |   | z |   | z + tz |
| 0 0 0  1 |   | 1 |   |   1    |

```

最后一列(tx,ty,tz)就是平移量。这样旋转、缩放、平移都用统一的矩阵乘法表示，GPU可以高效处理。

w分量的特殊含义：w=1表示点（可以被平移），w=0表示向量（方向，平移无意义）。在透视投影时，w≠1的值还用于透视除法，实现近大远小。

## MVP变换的完整流程

3D物体最终显示到2D屏幕，需要经过三个坐标空间的转换：

**Model（模型空间→世界空间）**
每个物体有自己的局部坐标系。比如一个立方体，顶点在(-0.5,-0.5,-0.5)到(0.5,0.5,0.5)之间。通过Model变换（TRS），把物体放到世界的正确位置。

**View（世界空间→观察空间）**
以相机为原点重新建立坐标系。想象相机在(10,5,0)看向原点，View变换把所有顶点转换到"以相机为中心"的坐标系。这样后续投影只需要处理相机前方的物体。

**Projection（观察空间→裁剪空间→屏幕空间）**
透视投影用4×4矩阵实现，把视锥体（金字塔形）压成长方体。之后做透视除法（除以w），映射到NDC（归一化设备坐标），最后视口变换到屏幕像素坐标。

为什么叫MVP：GPU顶点着色器的标准输出是经过MVP变换后的位置。理解这个流程是调试渲染问题的基础。

## TRS顺序为什么不可交换

Matrix4x4.TRS(position, rotation, scale)内部构建的矩阵是T×R×S，乘法从右往左应用。

**先缩放(S)**：顶点坐标乘以scale，物体大小变化，但还在原点周围。

**再旋转(R)**：绕原点旋转已经缩放的顶点。如果先旋转再缩放，缩放会沿着旋转后的轴向，导致非均匀缩放时形状扭曲。

**最后平移(T)**：把旋转后的物体移动到目标位置。如果先平移再旋转，物体会绕原点做圆周运动，而不是原地自转。

错误顺序的直观例子：先平移到(5,0,0)，再绕Y轴旋转90度，物体会从(5,0,0)移动到(0,0,5)，产生轨道运动而非自转。

## PLY文件解析的核心难点

PLY格式看似简单，但binary格式有几个陷阱：

**字节序问题**：little-endian vs big-endian。3DGS生成的PLY是little-endian，与x86架构一致，但跨平台时必须注意。

**Header解析的陷阱**：用StreamReader读取header时，它会预读缓冲区，导致FileStream的实际位置与预期不符。当切换到BinaryReader读取数据时，位置已经错位，读取的数据完全错误。

正确的做法：逐字节读取header，精确记录end_header后的文件偏移量，然后用BinaryReader从该位置开始读取。

**属性顺序**：3DGS的PLY有62个属性，必须按文件声明的顺序依次读取。即使只用x,y,z和颜色，也要跳过其他属性（法线、SH高阶项、透明度、缩放、旋转等），保持文件指针正确前进。

## 球谐函数(SH)的颜色表示

传统纹理是二维图像，只能表示"从正面看的颜色"。3DGS需要各向异性颜色——从不同方向看，颜色不同，模拟光照效果。

SH用基函数线性组合表示球面上的函数。第0阶是常数（基础颜色），第1阶是线性（类似漫反射），第2、3阶增加细节（类似镜面高光）。

3DGS存储45个SH系数（3通道×15个系数），但简化可视化时只用前3个（f_dc_0,1,2）。这些值范围约-2到+2，需要用Sigmoid函数映射到0-1的显示范围：

```

color = 1 / (1 + exp(-sh))

```

这比简单的(sh+0.5)更合理，因为SH值可能超出[-0.5,0.5]范围。

## 点云渲染的技术选择

**MeshTopology.Points**：告诉GPU把这些顶点当作独立的点绘制，而不是连成三角形。适合3DGS的高斯点云。

**IndexFormat.UInt32**：默认UInt16最多支持65535个顶点，3DGS需要百万级，必须用UInt32。

**Graphics.DrawMesh vs MeshRenderer**：MeshRenderer是组件模式，适合静态场景；DrawMesh是程序化调用，每帧可以更新矩阵和参数，适合动态点云。3DGS需要每帧根据相机位置排序点，所以用DrawMesh更灵活。

实现只加载了10万点（原文件600万点），因为全部加载会内存爆炸。后续需要用GPU Instancing或Compute Shader实现真正的百万级点云渲染。

## 坐标系转换的调试方法

模型"头朝下"或"侧躺"时，检查步骤：

1. 打印前几个顶点的坐标，确认数值范围是否合理
2. 检查x,y,z是否对应正确（有些软件是Z-up，Unity是Y-up）
3. 尝试坐标交换：`(x, z, -y)` 或 `(x, -z, y)` 等组合
4. 检查缩放方向，确保没有轴向反转

今天的自行车模型x坐标始终为0，是因为StreamReader缓冲导致数据读取错位，不是坐标系问题。这类问题需要通过打印原始数据来定位。

## 核心代码片段记忆

**精确定位PLY数据起始：**
```csharp
// 逐字节读header，避免StreamReader缓冲
while (ReadByte() != -1) {
    // 解析每一行
    if (line == "end_header") break;
}
long dataStart = fs.Position;  // 精确位置
fs.Seek(dataStart, SeekOrigin.Begin);
```

SH颜色解码：

```csharp
// Sigmoid映射，比线性映射更合理
float sigmoid(float x) {
    return 1.0f / (1.0f + Mathf.Exp(-x));
}
color = new Vector3(sigmoid(sh.x), sigmoid(sh.y), sigmoid(sh.z));
```

矩阵构建与应用：

```csharp
// TRS顺序：缩放→旋转→平移
Matrix4x4 m = Matrix4x4.TRS(pos, Quaternion.Euler(rot), scale);
Vector3 worldPos = m.MultiplyPoint3x4(localPos);
```
